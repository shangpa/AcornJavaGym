Inheritance

클래스 계층 구조

«abstract» Person
name, age, height, weight, phoneNumber
getBMI() · getBMICategory() · printInfo()*
├── Member extends Person
├── implements Registerable
├── implements Payable
└── implements FitnessEquipmentGuide
└── Trainer extends Person
├── implements Registerable
└── implements DietManageable

Person — 공통 속성 묶기

이름·나이·신체 정보 중복 제거. BMI 계산 로직을 한 곳에서 관리

printInfo() — 추상 메서드

서브클래스가 반드시 구현해야 함. 다형성의 핵심 진입점

다중 인터페이스

역할(결제/식단/기구안내)을 인터페이스로 분리해 Member와 Trainer의 책임을 명확히 구분

Abstraction

인터페이스 설계 — 추상화

«interface» Registerable
void register()
구현: Member, Trainer
«interface» Payable
void pay(int months)
구현: Member
«interface» DietManageable
void manageDiet(Member member)
구현: Trainer
«interface» FitnessEquipmentGuide
void explainEquipment(String bodyPart)
구현: Member

설계 의도: "무엇을 할 수 있는가"와 "어떻게 구현하는가"를 분리. 인터페이스를 타입으로 사용하면 구현 클래스를 교체해도 호출 코드가 변하지 않는다.

상속 (Inheritance)

Person → Member / Trainer
공통 필드·메서드 재사용으로 코드 중복 제거

다형성 (Polymorphism)

같은 printInfo() 호출이지만
Member냐 Trainer냐에 따라 출력 내용이 달라짐

추상 강제

abstract 선언으로 서브클래스가 반드시 자신만의 정보 출력을 구현하도록 강제

public abstract class Person {

  // 모든 필드를 private으로 은닉
  private String name;
  private String phoneNumber;
  private int    age;
  private double height;
  private double weight;

  // 제어된 접근: getter / setter
  public String getName()  { return name; }
  public void   setName(String n) { name = n; }
  ...
}

public class Trainer extends Person {

  private TrainerGrade grade = TrainerGrade.INTERN;

  // 외부에서 직접 등급 변경 불가
  // addManagedMember() → updateGrade() 경유
  private void updateGradeByMemberCount() { ... }
}

private 필드

모든 데이터 필드를 외부에서 직접 수정할 수 없도록 은닉

Getter / Setter

검증 로직이 필요한 경우 setter에서 처리. 읽기 전용이면 getter만 제공

내부 상태 보호

Trainer 등급은 외부 직접 세팅 불가. 오직 담당 회원 추가 시 자동 갱신

public void addManagedMember(Member m) {
  managedMembers.add(m);
  updateGradeByMemberCount(); // 자동 갱신
}

private void updateGradeByMemberCount() {
  int cnt = managedMembers.size();
  if      (cnt >= 10) grade = TrainerGrade.MASTER;
  else if (cnt >= 7)  grade = TrainerGrade.SENIOR;
  else if (cnt >= 4)  grade = TrainerGrade.JUNIOR;
  else               grade = TrainerGrade.INTERN;
}

Enum 활용

TrainerGrade를 Enum으로 선언해 허용되지 않은 등급값 입력 원천 차단

캡슐화 적용

updateGradeByMemberCount()는 private — 외부에서 임의로 등급 변경 불가

BMI 계산 & 식단 관리

// Person — BMI 공통 계산
public double getBMI(double h, double w) {
  return w / ((h / 100) * (h / 100));
}

// Trainer — DietManageable 구현
@Override
public void manageDiet(Member member) {
  double bmi = member.getBMI(
      member.getHeight(), member.getWeight());

  if      (bmi < 18.5) // 저체중
    System.out.println("고칼로리 단백질 식단 추천");
  else if (bmi < 22.9) // 정상
    System.out.println("균형 유지 식단 추천");
  else if (bmi < 25.0) // 과체중
    System.out.println("저칼로리 고단백 식단 추천");
  else               // 비만
    System.out.println("저탄고지 + 유산소 식단 추천");
}

저체중 (BMI < 18.5)

고칼로리 단백질 식단으로 근육량 증가 유도

정상 (18.5 ~ 22.9)

균형 잡힌 유지 식단 권장

과체중 (22.9 ~ 25.0)

저칼로리 고단백으로 체지방 감량

비만 (BMI ≥ 25.0)

저탄고지 식단 + 유산소 병행

@Override
public void pay(int months) {
  MembershipType plan;
  switch (months) {
    case 1:  plan = MembershipType.ONE_MONTH;     break;
    case 3:  plan = MembershipType.THREE_MONTH;   break;
    case 6:  plan = MembershipType.SIX_MONTH;     break;
    case 12: plan = MembershipType.TWELVE_MONTHS; break;
    default: return;
  }

  // 기존 만료일이 있으면 연장, 없으면 오늘부터 계산
  LocalDate base = (expirationDate != null)
      ? expirationDate : LocalDate.now();

  expirationDate_after_extension =
      base.plusMonths(plan.getMonths());

  System.out.println("결제 금액: "
      + plan.getPrice() + "원");
}

LocalDate 활용: java.time API로 날짜 계산을 간결하게 처리

정적 상수 패턴: MembershipType의 static final 인스턴스로 중복 객체 생성 방지

만료 연장: 기존 만료일 이후로 자동 연장하여 중복 결제 손해 없음

Summary

OOP 4대 원칙 적용 정리

원칙 적용 내용 코드 위치
🔒 캡슐화
(Encapsulation)		 모든 필드를 private으로 은닉, getter/setter로 제어된 접근 Person, Member, Trainer, MembershipType
🧬 상속
(Inheritance)		 Person → Member / Trainer. 공통 속성·메서드 재사용 model/Person.java → Member.java, Trainer.java
🎭 다형성
(Polymorphism)		 printInfo() 추상 메서드를 Member·Trainer가 각자 다르게 구현 Person.printInfo()* → 오버라이딩
🏗 추상화
(Abstraction)		 4개 인터페이스로 "할 수 있는 것"을 계약. 구현과 분리 interfaces/ 패키지 전체

설계 핵심: 인터페이스로 기능을 계약하고, 추상 클래스로 공통 상태를 묶으며, private 필드로 내부 상태를 보호하는 계층적 OOP 구조 완성
